01 · The system in one picture

Four parts, three workspaces, one tenant

Data flows top to bottom. An upstream job profiles PII; a factory on the analytics workspace builds two semantic models from that profile and the document corpus; an embedding job turns both into Vector Search indices; and a scope-routed agent serves them from the business-unit workspace. The YAML semantic models in Git are the human source of truth; the Delta tables are the runtime feed.

02 · Design principles

Four invariants a clone must keep

These are not features; they are the load-bearing constraints. They are why the system is defensible to a regulator and trustworthy to a business owner.

YML

Git-reviewable semantics

Both models are YAML in the repo. A change to a meaning, a PII flag, or a join is a reviewable diff. Delta is the runtime feed; YAML is the truth.

PII

Safe by construction

A sensitive column structurally cannot carry sample values, so no PII sample exists in any YAML, chunk, or index. Verified by a zero-leak assertion.

GRD

Grounded, never free-form

The agent retrieves real schema or document chunks, generates exactly one SELECT or cites real sources. Even build-time enrichment is grounded in VIB's glossary.

GOV

Governed and in-tenant

Approved Databricks models only; nothing leaves the tenant. Every SQL is read-only by AST proof. Every query is captured for SBV/NHNN and PDPL audit.

03 · The two semantic models

One executable, one a knowledge graph

Structured data needs an executable schema; documents need a graph. Forcing both into one shape would cripple both. They share vocabulary (models: [A, B], joinType) but only the structured side compiles to SQL.

04 · The agent

Scope is the route. Guards are not negotiable.

There is no LLM router: the UI scope dropdown is the route. One scope sends the turn down one of two plan-act-observe paths, both wrapped by the same shield, memory, and answer contract. Multi-scope and zero-scope are refused before any tool runs.

Proven, not promised. The refusal contract is an executable test suite (app/tests/red_team/): single-scope enforced before any tool touch, PII masked before the narrative prompt, injected DDL never reaching the warehouse, recalled SQL re-guarded, and one user unable to read another's session history.
05 · Embedding and Vector Search

One endpoint, indices split by scope

Both models are chunked into a single record schema and synced to four indices on one in-tenant Qwen3 endpoint. The split exists because a DELTA_SYNC index maps to exactly one source table and the SQL vector_search() the agent uses cannot filter by metadata: after dec_bu reached ~46% of chunks it was polluting FS retrieval, so each scope got its own table and index.

06 · Governance and security

Defensible at four layers

The governance story is the differentiator, and it is structural rather than a policy document. Each layer is independently testable.

1

Build-time PII safety

Sensitive columns never get a value dictionary, so no PII sample exists in YAML, chunk, or index. A zero-leak query asserts it.

2

Serving-time masking

Card / CCCD / phone / email masked at every LLM and persistence boundary, with Luhn+IIN so VND aggregates survive.

3

Read-only by AST proof

Every statement is one SELECT/WITH, no DDL/DML node, LIMIT clamped. The guard cannot be bypassed from the skill layer.

4

Identity and audit

Gated scope runs as the user (OBO) so Unity Catalog masks per user. Every query lands in system tables for SBV/NHNN and PDPL.

In-tenant, always. Agent (Claude Sonnet 4.6), factory (Claude Opus 4.8), and embeddings (Qwen3) are all Databricks Model Serving endpoints inside the VIB tenant. No content leaves to an external service, per PDPL and AI Law 134/2025.
07 · Deployment topology

Two bundles, one workspace each

Every job YAML is generated from a template + config; the config routes each job into the right bundle. Because each bundle targets exactly one workspace, a deploy can never leak a resource across workspaces.

08 · Cloning to a new business domain

Reusable core, thin domain layer

The expensive machinery is domain-agnostic. Standing the foundation up for a new domain means replacing a thin, well-bounded layer and keeping the core untouched. The pitch is not "we will build you a chatbot" but "we will point a proven, governed foundation at your data".

CORE

Clone as-is

The factory codegen + bundles, the chunker + Vector Search setup, the entire agent (soul / shield / skills / memory / answers), the web shell + UI, the SSE contract, the red-team suite, and the eval harness. None of these encode a domain; they encode the pattern.

DOMAIN

Replace per clone

The profiling scope, the structured + unstructured SMDL content, the golden glossary, the scope + routing maps, an index per new scope, the per-scope soul prompt, and the gold sets. Everything else is inherited, including the entire governance proof.

  1. Point PII profiling at the new catalog/schema so pii_tag_profile covers the new marts.
  2. Run the structured factory over the new scope to scaffold and enrich the model YAMLs; confirm the joins into the new dimension(s).
  3. Author the golden glossary for the domain's canonical concepts. This is the single highest-leverage artifact (A/B lift on canonical-term alignment: 2/10 to 9/10).
  4. Point the document factory (or the semantic-extract skill) at the new corpus; the knowledge-graph schema is unchanged.
  5. Add the scope to the UI dropdown and the SCOPE_INDICES / SCOPE_CATALOGS / scope_map; gate it with groups if access-controlled, exactly as rbdbb is.
  6. Add an index per new scope on the shared endpoint; the split already scales this way.
  7. Tune the per-scope soul for the domain's vocabulary and refusals.
  8. Write the gold sets and run the benchmark; budget time for the three known failure modes (aggregation grain, LIMIT, unit scaling).
